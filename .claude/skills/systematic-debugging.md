# Systematic Debugging Skill

수정 시도 전에 근본 원인을 찾는 체계적인 디버깅 가이드입니다.

## 핵심 원칙

> **근본 원인 조사 없이는 수정 금지**

Phase 1 조사 완료 전에는 어떤 솔루션도 제안하지 마세요.

## 4단계 필수 프로세스

### Phase 1: 근본 원인 조사

1. **에러 메시지 정독**
   - 전체 스택 트레이스 확인
   - 에러 코드, 라인 번호 파악

2. **일관된 재현**
   - 동일한 조건에서 반복 재현
   - 재현 단계 문서화

3. **최근 변경 확인**
   ```bash
   git log --oneline -10
   git diff HEAD~5
   ```

4. **컴포넌트 경계에서 증거 수집**
   - API 요청/응답 확인
   - 데이터베이스 쿼리 로그
   - 브라우저 콘솔 에러

5. **데이터 흐름 역추적**
   - 에러 발생 지점부터 입력까지 거슬러 올라감

### Phase 2: 패턴 분석

1. **동작하는 예제 찾기**
   - 비슷한 기능이 동작하는 코드 확인

2. **참조와 완전 비교**
   - 차이점 식별
   - 의존성 확인

### Phase 3: 가설 테스트

1. **구체적인 가설 수립**
   - "X가 Y 때문에 발생한다"

2. **최소한의 테스트**
   - 한 번에 하나의 변수만 변경
   - 결과 확인 후 다음 단계

### Phase 4: 구현

1. **실패하는 테스트 케이스 먼저 작성**
2. **근본 원인을 해결하는 단일 수정 적용**
3. **결과 검증**
4. **3회 이상 실패 시 아키텍처 문제 의심**

## 위험 신호 (즉시 중단)

다음 상황이 발생하면 중단하고 다시 시작:

- ❌ 조사 없이 수정 제안
- ❌ 동시에 여러 변경
- ❌ 수동 검증으로 테스트 생략
- ❌ 이전 시도 실패 후 또 다른 빠른 수정 시도

## 효과

| 접근법 | 해결 시간 | 첫 시도 성공률 |
|--------|-----------|----------------|
| 랜덤 수정 | 2-3시간 | ~40% |
| 체계적 디버깅 | 15-30분 | ~95% |

## 막혔을 때

여러 번 시도 후에도 해결되지 않으면:

> **이것은 아키텍처 문제 신호입니다. 또 다른 빠른 수정이 아닌 논의가 필요합니다.**

## 프로젝트 적용 예시

### 차트 렌더링 이슈

```typescript
// Phase 1: 증거 수집
console.log('Chart data:', chartData);
console.log('Config:', chartConfig);

// Phase 2: 동작하는 예제 비교
// 다른 차트 컴포넌트와 props 비교

// Phase 3: 가설 테스트
// "dataKey가 실제 데이터 구조와 불일치할 수 있다"
// → 데이터 구조 로깅으로 확인

// Phase 4: 단일 수정
<Bar dataKey={correctKey} />  // 올바른 키로 수정
```
